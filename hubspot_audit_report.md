# HubSpot Integration Audit Report

This document contains a full production-grade audit of the HubSpot CRM integration within Auricai.

## 1. CONNECTION FLOW (OAUTH / TOKEN)
**Result: 閥 FAIL (CRITICAL)**
* **Check**: HubSpot connection exists per organization, refresh token logic works, reconnect flow works.
* **Failure Point**: **Tokens are exposed to the frontend.** In `src/app/api/settings/route.ts`, the `GET` endpoint fetches `SettingsService.getOrgSettings`, which returns the raw `hubspot_connections` table record. This payload, containing the plaintext `access_token` and `refresh_token`, is actively served to the React client (`SettingsView.tsx` via SWR).
* **Warning**: Tokens are not encrypted at rest in the database (stored as standard text columns), which could violate strict vendor penetration tests.

## 2. PERMISSION VALIDATION
**Result: 泙 PASS**
* **Check**: The system safely validates and requests scopes `crm.objects.contacts.read`, `crm.objects.contacts.write`, `crm.objects.notes.read`, `crm.objects.notes.write`, and `crm.objects.deals.read` during connection.
* **Check**: 401 Unauthorized API responses are properly trapped in a try/catch block and bubbled back to the UI indicating a reconnect is required.

## 3. PUSH TO HUBSPOT WORKFLOW
**Result: 泯 WARNING**
* **Check**: Follows the complete step-by-step workflow (Email validation -> Contact Search -> HTML Note formatting -> Note Creation -> Note Association).
* **Failure Point**: **Incomplete/Draft Validation.** The backend (`HubSpotService.pushCaseStudyToHubSpot`) *does not* validate if `caseStudy.status === 'live'`. A malicious or buggy client request can push Draft case studies directly into the CRM.

## 4. DATA VALIDATION
**Result: 泯 WARNING**
* **Check**: Validates case study data before creating the Note.
* **Failure Point**: The system utilizes fallback UI text (like `"N/A"` for ROI or `"Success Story"` for Summary) instead of outright failing the request if a Case Study is fundamentally missing required fields (clientName, URL, ROI, etc.). This can lead to blank/useless notes polluting the CRM.

## 5. DUPLICATE PROTECTION
**Result: 泙 PASS**
* **Check**: A dedicated `hubspot_pushes` database table is implemented to record successful pushes (org_id, case_study_id, email). 
* **Check**: Both UI and Backend check this history lock before execution, preventing parallel or accidental identical pushes. 

## 6. ERROR HANDLING
**Result: 泙 PASS**
* **Check**: Missing contacts, invalid emails, and unauthorized errors all have distinct, graceful error messages.
* **Check**: Rate limits (HTTP 429) implement a defensive 1-time `2000ms` retry delay to automatically ingest the data.

## 7. UI INTEGRATION
**Result: 泙 PASS**
* **Check**: "Push to HubSpot" correctly configured only for `status === live` in the frontend tables and modals. 
* **Check**: The preview block matches the exact markdown generated by the `HubSpotService`.
* **Check**: The dedicated Analytics contextual banner behaves exactly as designed.

## 8. ANALYTICS CONNECTION

* **Check**: `DealRepository` strictly pulls HubSpot's Deal Amount dynamically into `sumAttributedPipeline` and strictly uses `stage === "closedwon"` into `sumVerifiableRevenue`. This fundamentally links CRM attribution to SaaS analytics.

## 9. DEAL LINKAGE SYSTEM
**Result: 泙 PASS**
* **Check**: The `DealAttributionRepository` elegantly handles mapping `external_deals` originating from HubSpot directly against Case Studies without crashing internal Deal tracking metrics. 

## 10. REAL-TIME BEHAVIOR

# AUDIT SUMMARY
### 閥 CRITICAL ISSUES
1. **OAuth Token Exposure:** The plaintext HubSpot `access_token` and `refresh_token` are sent directly to the client browser via the `/api/settings` JSON payload. These MUST be explicitly stripped out at the database retrieval or DTO layer before serialization.

### 泯 WARNING ISSUES
1. **Draft Push Breach:** The backend Server Action does not halt `pushToHubSpotAction` if `status !== 'live'`.
2. **Missing Field Validation:** Case studies missing key fields will still generate and push fallback data (`"N/A"`) instead of actively halting the API request. 
3. **Data At-Rest Security:** Tokens are stored as raw text strings without field-level cryptographic salting in the Postgres environment.

### 泙 MINOR ISSUES
* Minor inconsistencies where a missing contact does not prompt adding them automatically.
